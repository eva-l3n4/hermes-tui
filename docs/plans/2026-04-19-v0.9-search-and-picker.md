# v0.9 Plan: Conversation Search + Session Picker Overhauls

**Status:** Scoping notes — not yet scheduled. Captured 2026-04-19.

Three related-but-independent workstreams. Do them one at a time, in
whatever order feels right. They share a theme: **navigation** — finding
things, and moving back out to find something else.

---

## 1. Return to Picker (blocker for the others)

**Problem:** Once you enter a session, you're committed. No way to hop
back to the picker to switch sessions without killing and restarting
Kaishi. This is the gap Eva flagged.

**Approach:**
- New keybind (proposed: `Ctrl+B` for "back", or `Esc` from empty input)
- App state machine already has a `Picker` screen — need to re-enter it
  without tearing down the ACP connection entirely
- Question: do we keep the old session's state in memory for fast resume,
  or drop it? Probably drop — simpler, and the session persists ACP-side
  anyway.
- Refresh session list when returning (new sessions may have been created
  elsewhere, current one needs updated title/message count)

**Files touched:** `app.rs` (state transitions), `event.rs` (keybind),
`ui.rs` (status bar hint).

**Open questions:**
- Keybind collision check — `Ctrl+B` is free? confirm
- Should this be one-way (enter picker, lose current ACP session
  connection) or reversible? Reversible is nicer but more state.

## 2. Session Picker Overhauls

Riding along with #1 since we'll be in that code anyway.

**Candidates:**
- **Delete session** (`d` / `Delete` key, confirm modal) — already on the
  v0.9 wishlist. Needs ACP `session/delete` method support (check
  Hermes-side first).
- **Search / filter sessions** (`/` to filter by title or content
  preview) — picker is getting long, fuzzy filter would help.
- **Rename session** (`r` key, inline edit) — titles auto-generated, but
  sometimes you want to pin something memorable.
- **Sort toggle** — currently by recency; let user flip to alphabetical
  or by message count?
- **Archive / pin** — probably scope creep, flag for later.

**My pick for v0.9:** Delete + filter. Rename is nice but less urgent.

**Files touched:** `ui_picker.rs` (all of it, honestly), `event.rs`
(picker-mode key handling), maybe new `acp.rs` methods.

**ACP dependency:** Delete needs server-side support. Check
`hermes-agent` source for existing session management methods before
committing.

## 3. Conversation Search (Ctrl+F)

Independent of #1 and #2. Can ship first or last.

**Behavior:**
- `Ctrl+F` opens search bar overlay at bottom (above input)
- Type query → incremental match highlighting in rendered conversation
- `n` / `Enter` → next match (scroll to it)
- `N` / `Shift+Enter` → previous match
- `Esc` → close, clear highlights
- Match counter: `3/17` style

**Scope questions:**
- Plain substring or regex? Start plain, add regex later if we miss it.
- Case-sensitive? Smart-case (lowercase → case-insensitive, mixed →
  sensitive) is the vim/ripgrep default and feels right.
- Search scope: visible rendered text only, or raw markdown source? User
  expects visible text. Use what `ui.rs` produces after rendering.
- Match in tool call summaries and thinking blocks? Probably yes —
  they're part of the conversation.

**Implementation sketch:**
- New `SearchState` struct on `App`: query, matches (Vec<(turn_idx,
  line_idx, char_range)>), current index
- Rebuild match list on query change (debounced? probably unnecessary at
  our message volumes)
- `ui.rs` render pass takes `Option<&SearchState>` and applies highlight
  style to match ranges
- Scroll-to-match: compute which rendered line the match is on, set
  scroll offset

**Files touched:** new `ui_search_conv.rs` (to avoid conflict with
existing `ui_search.rs` which is Ctrl+R reverse history search), `app.rs`
(state), `event.rs` (Ctrl+F binding), `ui.rs` (highlight rendering).

**Pitfall:** `ui_search.rs` is already used for Ctrl+R. Name the new
module clearly to avoid confusion. Maybe rename existing to
`ui_history_search.rs` while we're at it.

## Suggested order

1. **Return to picker** (unlocks value of #2, small change)
2. **Conversation search** (highest daily-use payoff — ship this when
   energy is high)
3. **Picker overhauls** (delete + filter, depends on ACP check)

None of these require a breaking change. Target: roll them up as v0.9.0
when all three land, or ship incrementally as v0.8.2 / v0.8.3 / v0.9.0.
